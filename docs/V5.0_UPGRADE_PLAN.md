## 0. 升级前必读

`docs/V5.0_DESIGN_SOUL_APPENDIX.md` 是 V5.0 及后续版本升级的读前必看文档。任何 V5.0+ 任务开始前，必须先对照附录中的五条设计哲学、三个 wow moment 和 B 章反证案例；任何验收结论必须说明是否误伤了附录要保护的“氛围、内省、残响、未说之语和作家裁决权”。

如果本计划的任务拆分与附录的反证案例冲突，在用户没有明确更新文档前，以附录为准。尤其是：

- 不得再把分数当作文学质量裁判。
- 不得让工程诊断强迫氛围章节补冲突、行动和身体情绪。
- “采纳此改法”必须朝一键确认和 inline diff 收敛，不再制造正文目录辅助稿负担。
- 项目健康图必须避免红黄绿审判感，使用能表达热度、密度和趋势的卷轴式呈现。
- API/模型设置必须给出即时可理解的连通反馈。

## 1. 总定位

V5.0 不是单纯增加功能，而是把 V4.0 的“诊断和写作流水线”升级成两条并行闭环：

| 主线 | 目标 | 核心变化 |
|---|---|---|
| 文学保护内核 | 从“单一分数裁判”升级为“多视角观察 + 作家裁决” | score 不再代表文学质量；引入 ChapterMode、StyleProfile、LiteraryView、DiagnosticReservation 和 style_court |
| 写作伴侣体验 | 从“项目管理仪表盘”升级为“让作家想打开的写作伴侣” | 默认继续写、导航 9→4、单栏稿纸、页边批注、inline diff、后台任务、卷轴健康图 |

V5.0 的产品原则：

- 工程诊断只判断“管道是否漏水”和“AI 套话风险”，不替作家判断文学好坏。
- 文学批评保护“克制、残响、未说之语、自我欺骗和可被记住的瞬间”。
- 作家保留最终否决权，所有被保护或反驳的诊断不得再进入改稿 prompt。
- 写作时 UI 应当消失，AI 像“墙边的人”，只在需要时轻声提醒。

## 14. 明确不做

合并两个蓝图后，V5.0 不做以下事项：

- 不重写 RAG 引擎。
- 不整体重写 `novel_pipeline.py`、`llm_router.py` 等后端主流程；`quality_diagnostics.py` 只按蓝图增加 caveat、ChapterMode、语境敏感词表、开放式结尾和作家豁免，不推倒重来。
- 不替换四角色 LLM 路由。
- 不改 `chapter_ops.py` 删除流程。
- 不动 `versions/` 备份机制。
- 不做暗色模式，只预留 CSS 变量接口。
- 不做移动端适配。
- 不做实时协作。
- 不引入 React/Vue 等复杂前端框架。
- 不做语音输入或 AI 朗读。

## 15. 完成定义

V5.0 完成时，本项目应满足以下状态：

1. 打开 WebUI 后首先看到“今天”，并能直接继续最近章节。
2. 写作页默认是安静的单栏稿纸，而不是多面板驾驶舱。
3. AI 诊断以页边批注和可唤出面板存在，不常驻打扰。
4. score 只作为工程参考，不再充当文学裁判。
5. ChapterMode 和 StyleProfile 决定诊断阈值、套话容忍度和样本注入。
6. LiteraryView 能指出可记忆瞬间、未说之语、自我欺骗和文学风险。
7. style_court 能把工程诊断和文学保护之间的冲突分成 confirmed、contested、literary priorities。
8. 作家能对每条诊断采纳、保护或反驳，且选择会进入后续改稿链路。
9. 项目健康显示为工程稳健度、文学密度、风格一致度，不再合成单一总分。
10. Mock 模式不再假装评估文学性。
11. 长任务后台执行，用户可以继续写。
12. 全站 icon、字号、色板、空状态、错误提示和微动效统一到“写作伴侣”方向。

一句话验收：

> V5.0 的系统不再把作家拉进仪表盘，而是打开稿纸、保护文学性、把判断权交回给人。
